# Руководство по разметке `.docx`-шаблонов

> Для администраторов. Знание кода не требуется.

Это руководство объясняет, как из обычного Word-документа сделать шаблон для
автоматической генерации: где и как расставлять плейсхолдеры, как связать их с
формулами и ручными полями в редакторе, и как перенести 12 существующих шаблонов
из папки `шаблоны для первичного обращения`.

Генератор подставляет данные **прямо в исходный `.docx`** (движок
`buildDocxFromTemplate` на базе docxtemplater). Это значит, что **всё
форматирование Word сохраняется 1:1** — выравнивание, отступы, красная строка,
шрифты, интервалы, таблицы. Вы редактируете документ в Word как обычно и просто
вставляете плейсхолдеры там, где должны появиться данные.

---

## 1. Синтаксис плейсхолдеров

Все плейсхолдеры пишутся в **одинарных фигурных скобках** `{ ... }` прямо в тексте
Word. Источник истины по синтаксису — комментарий в шапке файла
`apps/backend/src/services/docxTemplateEngine.ts`.

| Тип | Синтаксис | Что подставляется |
|-----|-----------|-------------------|
| Формула (значение из CRM) | `{tagKey}` | Вычисленное значение формулы (данные сделки/контакта/компании) |
| Ручное поле | `{fieldKey}` | Значение, которое сотрудник вводит вручную в момент генерации |
| Цикл по товарам | `{#products}` … `{/products}` | Повторяющийся блок для каждой товарной позиции сделки |
| Картинка | `{%imageTag}` | Изображение из `data:image/...;base64,...` |

### 1.1. Формулы — `{tagKey}`

`tagKey` — это короткий латинский ключ формулы (например `client_fio`,
`policy_number`, `incident_date`). Вы придумываете формулу в редакторе, а ключ
система предлагает автоматически (слугифицирует из названия). В Word вы пишете
ровно `{client_fio}` там, где должно встать ФИО клиента.

Пример строки в Word:

```
Потерпевший: {client_fio}, дата рождения {client_birthdate}
```

### 1.2. Ручные поля — `{fieldKey}`

Ручное поле — это то, что **нельзя посчитать из CRM** и сотрудник заполняет руками
при генерации (тип поля: текст / многострочный текст / число / дата). Синтаксис тот
же, что у формул — `{fieldKey}`:

```
Дополнительные пояснения: {extra_notes}
```

> **Важно про коллизии ключей.** Если `fieldKey` совпадёт с `tagKey` формулы,
> приоритет у формулы — ручное поле будет проигнорировано. Ручные поля только
> «закрывают» те ключи, которых нет среди формул. Поэтому давайте формулам и полям
> разные ключи.

### 1.3. Товарные позиции — `{#products}` … `{/products}`

Если в документе есть таблица товаров (позиции сделки), оберните строку-шаблон
таблицы в секцию цикла. Всё между `{#products}` и `{/products}` повторится для
каждой позиции.

Доступные поля внутри цикла (заполняются автоматически, **их не нужно** привязывать
в редакторе — они зарезервированы):

| Поле | Значение |
|------|----------|
| `{INDEX}` | Порядковый номер строки (с 1) |
| `{PRODUCT_NAME}` | Наименование товара |
| `{PRICE}` | Цена за единицу |
| `{QUANTITY}` | Количество |
| `{MEASURE_NAME}` | Единица измерения |
| `{DISCOUNT_SUM}` | Сумма скидки |
| `{TAX_RATE}` | Ставка налога |
| `{SUM}` | Сумма по строке |
| `{SORT}` | Сортировка |
| `{PRODUCT_ID}`, `{ID}` | Идентификаторы |

Как разметить строку таблицы в Word (одна строка-образец):

```
| {#products}{INDEX} | {PRODUCT_NAME} | {QUANTITY} {MEASURE_NAME} | {PRICE} | {SUM}{/products} |
```

`{#products}` ставится в начало первой ячейки строки-образца, `{/products}` — в
конец последней ячейки той же строки. docxtemplater размножит всю строку таблицы.

### 1.4. Картинки — `{%imageTag}`

Изображения помечаются префиксом `%`: `{%logo}`. Значение должно быть
`data:image/...;base64,...` (png/jpg/jpeg/gif/svg/webp/bmp). Внутри цикла товаров
доступны картинки позиции:

- `{%PREVIEW_PICTURE_BASE64}` — превью товара,
- `{%DETAIL_PICTURE_BASE64}` — детальное изображение.

Эти теги тоже зарезервированы (заполняются движком автоматически, привязывать не
нужно). Размер по умолчанию — 150×150 px.

## 2. Привязка формул и полей в редакторе (UI)

После того как документ размечен плейсхолдерами, шаблон загружается в приложение, и
плейсхолдеры нужно связать с формулами/полями. Последовательность:

1. **Загрузите `.docx`.** На странице загрузки шаблона выберите размеченный файл и
   тему. При загрузке backend сканирует документ и собирает список всех найденных
   плейсхолдеров (`scanDocxPlaceholders`).
2. **Откройте редактор шаблона.** Панель **«Теги шаблона»** показывает каждый
   найденный тег и его статус:
   - **formula** — тег привязан к формуле;
   - **field** — тег привязан к ручному полю;
   - **reserved** — служебный тег товарного цикла/картинки
     (`products`, `INDEX`, `PRODUCT_NAME`, `SUM`, `{%PREVIEW_PICTURE_BASE64}` и
     т.д.) — заполняется автоматически, привязка не требуется;
   - **unbound** — тег есть в документе, но ещё не связан ни с формулой, ни с
     полем. Такой тег при генерации останется пустым.
3. **Создайте формулу** через диалог **FormulaBuilder**: задайте название (из него
   предложится `tagKey`), напишите выражение (mathjs) с токенами вроде `DEAL.X`,
   `CONTACT.Y`, `COMPANY.Z`, при необходимости проверьте значение на тестовой
   сделке. Сохранённая формула добавляется в массив `formulas[]` шаблона, а её ключ
   и есть тот `{tagKey}`, который вы вписали в Word.
4. **Создайте ручное поле** через диалог **ManualFieldBuilder**: название, ключ,
   тип (text / textarea / number / date), флаг «обязательное», подсказка. Поле
   добавляется в массив `fields[]`. Его ключ — `{fieldKey}` из Word.
5. **Добейтесь, чтобы не осталось `unbound`-тегов** (кроме намеренно пустых).
   Сохраните шаблон.

> Плейсхолдеры размещаются **в Word**, а не в редакторе. Редактор не вставляет
> теги в текст — он только хранит формулы/поля и связывает их с уже
> существующими в `.docx` тегами по совпадению ключа.

## 3. Миграция 12 существующих шаблонов

Папка `шаблоны для первичного обращения` содержит 12 готовых Word-документов, в
которых данные обозначены **человекочитаемыми заглушками**, а не плейсхолдерами:

- `(автомат из КК)`, `автомат из КК`
- `тянется автоматически из справочника` / `из профиля ответственного юриста`
- `ХХХ`, `XX`, `XXX`
- `_____` (подчёркивания-пропуски)

Эти заглушки нужно **заменить на теги** `{tagKey}` / `{fieldKey}`. Сам перенос
выполняется **вручную в Word** — это работа администратора.

### Пошагово для одного шаблона

1. Откройте документ в Word.
2. Пройдите по тексту и для каждой заглушки решите:
   - данные **есть в CRM** (сделка/контакт/компания) → это **формула** →
     поставьте `{tagKey}` (например `(автомат из КК)` рядом с ФИО → `{client_fio}`);
   - данные **вводятся руками** при генерации → это **ручное поле** → поставьте
     `{fieldKey}`;
   - данные из **справочника** (название/адрес СК, контакты юриста) → формула,
     ссылающаяся на нужный источник, → `{tagKey}`.
3. Если есть таблица позиций — оберните строку-образец в `{#products}…{/products}`
   и расставьте поля цикла (`{PRODUCT_NAME}`, `{QUANTITY}`, `{SUM}` …). Лишние
   строки-примеры удалите, оставив одну строку-образец.
4. **Сохраните форматирование**: не меняйте выравнивание и отступы — просто
   замените текст заглушки на тег. Шапка (справа), заголовок (по центру), основной
   текст (по ширине), красная строка и интервалы останутся как есть — движок их не
   трогает.
5. Сохраните `.docx`, загрузите его в приложение и привяжите формулы/поля
   (раздел 2). При повторной правке разметки шаблон нужно **перезагрузить** заново.

### Пример замены (шаблон «первичное заявление в СК ПВУ»)

| Было (заглушка в Word) | Стало (тег) | Тип |
|------------------------|-------------|-----|
| `Наименование Страховой компании (автомат из справочника)` | `{insurer_name}` | формула (справочник СК) |
| `Адрес СК (тянется автоматически из справочника)` | `{insurer_address}` | формула (справочник СК) |
| `ФИО заявителя (тянется автомат из КК)` | `{client_fio}` | формула (контакт) |
| `Дата и время страхового случая: автомат из КК` | `{incident_datetime}` | формула (сделка) |
| `Паспорт: серия номер дата выдачи кем выдан (автомат из КК)` | `{client_passport}` | формула (контакт) |
| `по полису ХХХ (автомат из КК)` | `{policy_number}` | формула (сделка) |
| `Государственный регистрационный знак: автомат из КК` | `{vehicle_plate}` | формула (сделка) |
| (поле, которого нет в CRM) | `{extra_notes}` | ручное поле |

Ключи в таблице — примерные; используйте те, что предложит редактор при создании
формулы/поля, и впишите ровно такие же в Word.

## 4. Чек-лист приёмки шаблона

Перед тем как считать шаблон готовым, проверьте:

- [ ] Все заглушки в документе заменены на теги `{...}` (не осталось «автомат из
      КК», «ХХХ», «_____»).
- [ ] В панели «Теги шаблона» **нет `unbound`-тегов** (кроме намеренно пустых).
- [ ] Каждый `{tagKey}` связан с формулой, каждый `{fieldKey}` — с ручным полем.
- [ ] Товарная таблица (если есть) обёрнута в `{#products}…{/products}`, и поля
      цикла берутся из зарезервированного набора.
- [ ] **Preview совпадает с Word**: предпросмотр на странице генерации визуально
      повторяет исходный документ.
- [ ] **Сгенерированный `.docx` открывается в Word с форматированием 1:1**:
      выравнивание шапки (справа), заголовка (по центру), основного текста (по
      ширине), отступы / красная строка / интервалы сохранены.
- [ ] На тестовой сделке с заполненными полями и товарами все значения
      подставились корректно.

## Справочник: зарезервированные теги товарного цикла

Эти теги **нельзя** использовать как ключи формул/полей — они заняты движком
(см. `apps/frontend/src/lib/templateTags.ts`, `RESERVED_PRODUCT_TAGS`):

`products`, `INDEX`, `PRODUCT_NAME`, `PRICE`, `QUANTITY`, `DISCOUNT_SUM`,
`TAX_RATE`, `SUM`, `MEASURE_NAME`, `SORT`, `PRODUCT_ID`, `ID`,
`PREVIEW_PICTURE_BASE64`, `DETAIL_PICTURE_BASE64`, `MORE_PHOTO_BASE64`.
